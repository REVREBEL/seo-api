# Schema Design

## Primary keys

Prefer compact, ordered keys for large relational tables:

```sql
CREATE TABLE user_account (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  email TEXT NOT NULL UNIQUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Guidelines:

- Prefer `BIGINT GENERATED ALWAYS AS IDENTITY` for internal primary keys.
- Avoid random UUIDv4 primary keys on high-write, large tables because they increase B-tree fragmentation and storage.
- If UUIDs are required, prefer time-ordered UUIDs generated by an approved client library or an explicitly available extension. Verify function availability on the target PostgreSQL version before recommending a database-side UUID function.
- Use the same data type for foreign keys as the referenced primary key.

## Data types

| Prefer | Avoid by default | Reason |
| --- | --- | --- |
| `TEXT` | arbitrary `VARCHAR(n)` limits | PostgreSQL stores them similarly; use constraints for real rules. |
| `TIMESTAMPTZ` | `TIMESTAMP` without time zone | Safer for distributed applications and audit trails. |
| `JSONB` with constraints/indexes | unvalidated JSON blobs | Flexible without abandoning data quality. |
| `BIGINT` / `INTEGER` | provider-specific or extension-only types | Portable and predictable. |
| `CHECK` constraints | `ENUM` for frequently changing states | CHECK constraints are easier to alter. |

Example constrained status:

```sql
CREATE TABLE orders (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  status TEXT NOT NULL CHECK (status IN ('pending', 'paid', 'shipped', 'cancelled')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

## Foreign keys

PostgreSQL does not automatically create indexes on foreign key columns. Create them explicitly:

```sql
CREATE TABLE orders (
  id BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  customer_id BIGINT NOT NULL REFERENCES customer(id) ON DELETE CASCADE
);

CREATE INDEX orders_customer_id_idx ON orders (customer_id);
```

Guidelines:

- Index every foreign key used for joins, deletes, or parent-table updates.
- Avoid circular dependencies unless there is a clear domain reason.
- State `ON DELETE` behavior explicitly: `RESTRICT`, `CASCADE`, `SET NULL`, or `NO ACTION`.
- Add `NOT NULL` where the domain requires a value.

## Naming conventions

- Tables: singular `snake_case` (`user_account`, `order_item`).
- Columns: singular `snake_case` (`created_at`, `customer_id`).
- Indexes: `{table}_{column_or_purpose}_idx`.
- Constraints: `{table}_{column_or_purpose}_{type}`.

## Normalization and denormalization

- Start normalized for correctness and flexibility.
- Denormalize only for proven hot read paths.
- When denormalizing, document the source of truth and refresh mechanism.
- Use materialized views or summary tables only when the refresh cadence and staleness tolerance are explicit.
